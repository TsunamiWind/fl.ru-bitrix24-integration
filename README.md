# FL.ru ↔ Bitrix24 Мост

Двусторонняя синхронизация сообщений между биржей фриланса FL.ru и порталом Bitrix24. Когда клиент пишет вам на FL.ru — создаётся чат в Bitrix24, и вы отвечаете прямо оттуда, не заходя на фриланс-биржу.

## Возможности

- **Сообщения с FL.ru → Bitrix24:** новые сообщения от клиентов (отклики на проекты, сделки, типовые услуги, личные сообщения) автоматически появляются в Bitrix24
- **Ответы из Bitrix24 → FL.ru:** ваш ответ в чате Bitrix24 отправляется клиенту на FL.ru
- **Раздельные чаты:** каждое обсуждение на FL.ru — отдельный чат в Bitrix24, без дублирования
- **Имена отправителей:** бот подписывает сообщения именем клиента с FL.ru
- **Только новые сообщения:** старые переписки не импортируются — синхронизируются только сообщения, пришедшие после запуска моста
- **Пересылка файлов:** изображения и документы передаются в обе стороны
- **Индикатор доставки:** после успешной отправки ответа на FL.ru под сообщением в Bitrix24 появляется ❤ (лайк)

## Поддерживаемые типы диалогов FL.ru

| Тип | Источник | Как выглядит в Bitrix24 |
|-----|----------|------------------------|
| Отклики на проекты | Клиенты, ответившие на вашу заявку | Чат «FL.ru: Название проекта» |
| Безопасные сделки | Действующие заказы | Чат «FL.ru Order: Имя клиента» |
| Типовые услуги | Покупатели услуг из магазина | Чат «FL.ru Service: Имя» |
| Личные сообщения | Прямые сообщения FL Team | Чат «FL.ru: Имя» |

## Как это работает

```
FL.ru клиент пишет вам
        │
        ▼
  ┌───────────┐    каждые 30 сек    ┌──────────────┐
  │   FL.ru   │ ◄────────────────── │  bridge.py   │
  │           │ ──────────────────► │              │
  └───────────┘   проверка новых    └──────┬───────┘
                    сообщений              │
                                           │ im.chat.add
                                           │ imbot.v2.Message.send
                                           ▼
                                    ┌──────────────┐
                                    │  Bitrix24    │
                                    │  Чат с ботом │
                                    └──────┬───────┘
                                           │
                    каждые 5 сек           │
                  проверка ответов         │
                                           ▼
  ┌───────────┐   POST на API    ┌──────────────┐
  │   FL.ru   │ ◄─────────────── │  bridge.py   │
  │  ответ    │                  │              │
  └───────────┘                  └──────────────┘
```

## Требования

- Python 3.11+
- Доступ к платному тарифу Bitrix24 (REST API)
- Аккаунт на FL.ru

## Установка

```bash
git clone https://github.com/TsunamiWind/fl.ru-bitrix24-integration.git
cd fl.ru-bitrix24-integration
pip install -r requirements.txt
```

## Настройка

### 1. Bitrix24

1. В левом меню Bitrix24 откройте **Приложения → Разработчикам**
2. Вкладка **Готовые сценарии → Другое → Входящий вебхук**
3. Назовите вебхук (например, «FL.ru Bridge»)
4. В **Настройках прав** включите:
   - **Чаты** (`im`)
   - **Чат-боты** (`imbot`)
   - **Пользователи** (`user`)
5. Сохраните и скопируйте URL вебхука
6. Запустите скрипт для регистрации бота:

```bash
python setup_bot.py
```

Либо зарегистрируйте бота вручную через вебхук (см. раздел «Регистрация бота» ниже).

### 2. FL.ru

1. Войдите на FL.ru в браузере
2. Откройте **Инструменты разработчика** (F12) → вкладка **Application / Приложение** → **Cookies**
3. Скопируйте значения следующих cookie в файл `config.json`:
   - `PHPSESSID`
   - `id`
   - `pwd`
   - `XSRF-TOKEN`
   - `name`
   - `user_device_id`
4. Значение `csrf_token` равно `XSRF-TOKEN`
5. Укажите `your_user_id` — это ваш числовой ID на FL.ru (из cookie `id`)

### 3. config.json

Скопируйте шаблон и заполните своими данными:

```bash
cp config.example.json config.json
```

Отредактируйте `config.json`, заполнив поля учётными данными из шагов выше.

## Запуск

```bash
python main.py
```

Мост запускает два фоновых цикла:
- **Каждые 30 секунд** — проверка новых сообщений на FL.ru
- **Каждые 5 секунд** — проверка ваших ответов в Bitrix24

При первом запуске для каждого существующего диалога на FL.ru записывается «базовый уровень» (максимальный ID сообщения). Сообщения, пришедшие до этого момента, не импортируются. Когда клиент напишет новое сообщение — будет создан чат в Bitrix24 и сообщение синхронизируется.

## Регистрация бота (если бот ещё не создан)

Мост ожидает, что чат-бот уже зарегистрирован в Bitrix24. Для регистрации вызовите метод REST API:

```
POST /rest/1/ваш-webhook-код/imbot.v2.Bot.register.json
{
    "fields": {
        "code": "fl_ru_bridge",
        "botToken": "ваш_токен_бота_до_40_символов",
        "type": "personal",
        "isHidden": true,
        "properties": {
            "name": "FL.ru Bridge",
            "workPosition": "Message relay from fl.ru"
        }
    }
}
```

Скопируйте `botId` из ответа и `botToken`, который вы указали, в `config.json`.

## Поддержание сессии FL.ru

Cookie `PHPSESSID` на FL.ru истекает примерно через 5 дней бездействия. Признак истечения — HTTP 403 при запросе к API.

**Как обновить:**
1. Войдите в FL.ru в браузере заново
2. Скопируйте новый `PHPSESSID` из Cookie (и новый `XSRF-TOKEN`, если изменился)
3. Обновите значения в `config.json`
4. Перезапустите мост

## Ограничения

- **1 000 символов** — максимальная длина сообщения в откликах на проекты на FL.ru. Более длинные ответы автоматически обрезаются.
- **Без реального времени** — сообщения синхронизируются с задержкой до 30 секунд.
- **Только текст и файлы** — стикеры, реакции, опросы не передаются.
- **Файлы из Bitrix24 → FL.ru** — загружаются через публичный CDN fl.ru как ссылки в тексте сообщения. Прямая загрузка вложений через современный API блокируется защитой браузера (`Sec-Fetch-Site`).
- **Одна учётная запись FL.ru** — мост работает от имени одного пользователя.

## Безопасность

- `config.json` содержит чувствительные данные (cookie, webhook-секрет). **Никогда не коммитьте его.**
- Webhook-секрет Bitrix24 даёт доступ ко всем разрешённым методам API от вашего имени.
- Cookie FL.ru эквивалентны полному доступу к вашему аккаунту.
- Файл `bridge.db` содержит только технические ID сообщений, без содержимого переписки.

## Решение проблем

Если что-то работает не так — при создании issue на GitHub приложите файл `log.txt`. В него записываются все запросы к API, ответы сервера и ошибки — это поможет быстро найти причину.

Сам лог-файл не содержит паролей или cookie (только URL запросов и обрезанные ответы), но перед отправкой убедитесь, что в нём нет конфиденциальных данных.
